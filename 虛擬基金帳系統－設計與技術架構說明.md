# 虛擬基金帳（Virtual Fund Ledger）系統設計說明

> 目的：將「實體銀行帳戶」與「資金用途（基金）」解耦，透過虛擬基金帳與交易帳本（Ledger）來解決混帳、難以追蹤的問題。

---

## 一、系統定位與性質

### 1. 這是一個輕量系統嗎？

**是的，在初期它是典型的輕量系統（Lightweight System）**，原因如下：

- 使用者數量少（個人 / 夫妻）
- 資料量小（交易筆數有限）
- 沒有即時高併發需求
- 無金流實際扣款（純記帳 / 邏輯帳）

但在「設計思想」上，它其實接近：

> **簡化版的 Accounting / Ledger System**

所以：
- 初期可以非常輕
- 結構上要「可成長」

---

## 二、核心設計理念

### 關鍵抽象

- **Account（實體帳戶）**：錢實際放哪（不負責用途）
- **Fund（基金）**：錢的用途與規則（不等於帳戶）
- **Transaction（交易帳本）**：唯一的事實來源（Source of Truth）

> 所有餘額、報表、圖表，**一律由 Transaction 計算得出**

---

## 三、資料模型（Core Schema）

### 1. Fund（基金）

- id
- name
- description
- monthly_amount
- cycle_type (monthly)
- cycle_day (1 ~ 31)
- is_active

---

### 2. Account（實體帳戶）

- id
- name (薪轉戶 / 空戶頭)
- type (salary / saving)

---

### 3. Transaction（交易帳本）

- id
- fund_id
- account_id
- type (TOP_UP / EXPENSE / ADJUST / TRANSFER)
- amount (+ / -)
- category (optional)
- description
- occurred_at

---

## 四、餘額與邏輯規則

### Fund Balance 計算方式

```
Fund Balance = SUM(Transaction.amount WHERE fund_id = ?)
```

⚠️ **不儲存 fund_balance 欄位**，避免資料不一致。

## 五、技術架構選型（分階段）

### Phase 1：個人 / 夫妻使用（最小可行）

#### 推薦組合 A（你會很快上手）

- Frontend：Vue 3
- Backend：Supabase
  - Auth
  - PostgreSQL
  - Row Level Security
- Scheduler：Supabase Cron
- Charts：ECharts / Chart.js

**優點**：
- 極低維運成本
- 快速開發
- SQL + RLS 很適合 Ledger 類系統

---

### Phase 2：輕量公開（小量使用者）

#### 推薦組合 B（稍微正式）

- Frontend：Vue 3 / Nuxt
- Backend：
  - Node.js (Fastify / NestJS)
- DB：PostgreSQL
- Auth：JWT / OAuth
- Scheduler：
  - node-cron
  - 或 DB job

**特點**：
- API 明確
- 可部署在 Fly.io / Railway / Render

---

### Phase 3：正式 SaaS 化（成長型）

#### 推薦組合 C（可擴充）

- Frontend：Nuxt
- Backend：NestJS
- DB：PostgreSQL
- Queue：BullMQ / Redis
- Scheduler：Queue-based Job
- Analytics：Materialized View

---

### 目前決策（Phase 2.5）

基於目前專案目標（小量使用者、快速交付、但需保留成長彈性），
本專案正式採用 **Phase 2.5** 作為實作策略：

- **技術方向**：以 Phase 2 為主體落地，並預留 Phase 3 升級邊界
- **開發規範基準**：以 `CLAUDE.md` 為唯一開發規範依據，並可依需求滾動更新
- **開發原則**：採 **async-first**，排程、通知、重查詢、批次任務預設走非同步處理

簡單來說：
- 現階段先追求「快速可用」
- 架構上同時確保「未來可擴充、可演進」

---

## 六、為什麼 Ledger 架構適合公開產品

- 每一筆錢「可回溯」
- 不怕資料壞帳
- 天生支援報表與圖表
- 可加審計（Audit Log）

這也是為什麼：
- 銀行
- 會計系統
- 金融 SaaS

都用 Ledger 思維。

---

## 七、未來擴充方向（你現在先不用做）

- 多使用者（夫妻共用）
- 權限（只讀 / 可編輯）
- 基金預算警示
- AI 消費分析（支出趨勢、超支預測）
